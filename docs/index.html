Loading Audio Players

Mac instructions

What this does: erases the USB audio players plugged into your Mac and loads your audio content onto all of them at once. You choose exactly which disks are erased, and you confirm before anything happens.

⚠️ It erases whole DISKS. If a disk has several partitions or volumes on it, all of them are wiped together. Unplug any drive you do not want erased (backup drives, etc.) before you start — the chooser also lets you deselect them.
ℹ️ The devices are kept free of hidden macOS junk (.DS_Store, ._*). The script cleans up after itself and leaves your Mac's settings and your content folder exactly as it found them.
📥 Get the loader: Download the latest release. It's a tiny universal Mac script — it runs on both Apple Silicon and Intel Macs, with nothing to install. You add your own numbered audio-content folders next to it (see the Player-Specific Setup tab).

Steps

  1. Put the loader in your content folder

    Unzip the download to get load_content.sh. Then put that file inside your content folder — the one that holds your numbered audio folders (001 …, 002 …), exactly as shown below. The loader copies whatever sits in its own folder onto the devices, so it has to live right beside your audio.

    (If someone handed you a ready-made folder, load_content.sh is already inside it — just open that folder.)

  2. Plug in the devices

    Plug in the USB hub with the devices to be loaded.

  3. Open Terminal

    Press Cmd + Space, type terminal, and press Return. (Terminal is a built-in Mac app. A small window opens.)

  4. Type bash and a space

    In the Terminal window, type the word bash followed by one space. Do not press Return yet.

  5. Drag the script onto Terminal

    Drag the load_content.sh that is inside your content folder onto the Terminal window and drop it — that exact copy, the one sitting beside your audio, is what decides what gets loaded. Its location appears after the word bash.

    Can't drop it on the Terminal window? If Terminal is hidden behind other windows — or you're looking at all your windows at once (like the picture below) — don't drop the file into empty space. Instead: drag load_content.sh onto Terminal's icon in the Dock (the row of app icons along the bottom of the screen) — or onto its window thumbnail — and keep holding, don't let go. After a second the Terminal window jumps to the front; now slide the file onto that window and release.

  6. Press Return

    Click the Terminal window and press Return.

  7. Choose a volume label

    A box asks for the volume label — the name the devices will have. Click OK for the default (PLAYER), or type your own (letters and numbers, up to 11).

    💡 Tip: use a new, unused name each batch — it makes the after-the-run device check (see Verify every device, below) reliable.

  8. Confirm the device count

    A message tells you how many external disks were found, with buttons Cancel / Show in Finder / Choose disks…. "Show in Finder" opens the drives so you can peek at what's on them; the message then comes back. Click Choose disks… to continue.

  9. Pick which disks to load

    A list appears. Each row is one DISK, shown with its size and the names of all volumes on it. All disks start selected. If you see a disk you want to keep (for example a backup drive), hold Cmd and click that row to deselect it. Then click Continue….

  10. Read and confirm

    A final message shows exactly which disks will be ERASED and which will be KEPT. Read it! Then click Erase & Load (or Back to change the selection).

  11. Wait

    A progress bar shows overall progress and time remaining. As each device finishes you'll see a line like:

      OK    disk5 loaded and ejected — safe to unplug

    You may unplug that device right away. The output is colour-coded — green for devices that loaded, red for failures. At the end you get a RESULTS summary grouped into GOOD CHECK REDO (see below), and a dialog asking Run again? — click it to do the next batch (or redo failures), or Done to stop.

🔔 You don't have to watch the screen. Live progress also shows in the Terminal tab/window title (visible even when Terminal is in the background or the Dock), and when the whole batch finishes a notification banner with a sound announces the result — e.g. "18 loaded, 2 need redo." So you can start the run and step away.
To load more devices: plug in the next batch and repeat from step 4. (Press the ↑ Up-Arrow key in Terminal to repeat the previous command, then Return.)

When some devices fail

This is normal with cheap USB devices. Bad cables, loose connectors, corrosion, or humidity make some devices fail — expect it. The RESULTS summary at the end sorts every device:

GroupMeaning
GOOD Loaded, ejected, and safe to pull. These have dropped off the Mac, so they will not appear if you run again.
CHECK The content copied fine, but something needs a look (for example it could not eject — just unplug it by hand).
REDO Not loaded — retry these. They are still connected and have been renamed to "REDO", so you can spot them.

Easiest way to finish a batch: when the run ends, a dialog asks Run again? Unplug the finished devices and set them aside; leave the blinking failures plugged in (sorting details below). Add a fresh batch of devices too if you like, then click Run again. Because the finished devices have already ejected and detached, only the still-connected REDO failures — plus anything new you added — show up in the list, so it redoes just the failures. Repeat until nothing is left blinking. The same dialog appears after a successful run too — swap in the next batch and click Run again, or Done to stop.

Which physical device failed?

When the run finishes with failures, the script helps you find them two ways, automatically:

1. It blinks the failed devices' lights — so leave those plugged in. A pop-up lists the failures and blinks their lights so you can spot them in the hub. Do the opposite of unplugging them: take out the devices that are not blinking and sort them into piles by their light:

  • Steady / solid light — finished. These loaded and were ejected. Unplug them and set them aside as your done pile. (Still test them later — see Verify every device, below.)
  • No light at all (dark) — dead battery. The device switched off or its battery died mid-copy, so it failed too. Unplug it and set it aside to charge fully before you reload it.
  • Blinking — failed; leave it connected. This is a REDO device and stays for the next round.

Your hub now holds only the still-connected blinking devices. Plug in a fresh batch of new devices now too if you want, then click Run again — it redoes the blinking failures (and loads any new ones). Repeat until nothing blinks. Choose Done when you're finished.

⚠️ Also watch for dark devices. A device whose battery died or that switched off mid-copy shows no light at all — not blinking. It failed too and must be redone. A steady/solid light means the device finished — but still check its content before distributing.

Charge a dark device before reloading it. The loading cable and hub may carry data only and not deliver a charge, so simply plugging a dead device back in may not wake it. Charge it fully first — in the sun, or with a proper charging cable — then reconnect it to reload.
🔧 For a device that keeps failing: before giving up on it, re-seat its cable — unplug and firmly reconnect both ends, in case a connector is loose or not fully seated. For stubborn cases, a little electrical contact cleaner on the connector can help, and so can a full charge — a low battery alone can make a copy fail.

2. It marks them in Finder — and macOS helps you avoid pulling the wrong one. The failed devices stay mounted and are renamed REDO. Open a Finder window and watch the list under your Mac's name (e.g. "Seth's MacBook Pro") as you unplug — the only USB volumes still shown there are the failed REDO ones.

  • The finished devices were already ejected, so pulling one is silent — nothing complains.
  • If you pull a device and macOS warns “Disk not ejected properly” (unsafe removal), you just took out a still-mounted REDO device — one you meant to leave connected. No harm done, but the warning tells you it belongs in the failures pile: put it back, or set it aside to redo.
  • To pin down which physical device is which, unplug them one at a time and watch for a REDO to disappear from the Finder list.

Verify every device before handing it out

This tool cannot catch every failure. A device that fully disconnected or lost power partway through can still be reported as loaded, and a blank or half-written device is not always obvious. So test the devices — even when the run reports no failures:

  1. Pull the failed (REDO) devices first (see above), so they're out of the way.
  2. Unplug and re-plug the rest — or just re-plug the whole hub — and confirm each one comes back mounted with the label you chose (e.g. PLAYER).
  3. One device at a time is the most reliable check.
💡 Use a new, unused label for each batch. Then a device that quietly kept its old content — and never really got written this run — is obvious: it comes back with the wrong name, or doesn't mount at all.

Troubleshooting

MessageWhat it means / what to do
Skipping diskN — WRITE-PROTECTED That device (or card) has a hardware write-protect lock. Unlock it and run again.
SKIPPED diskN — device changed A drive was plugged/unplugged after the list was shown. Nothing was erased on it. Run again for a fresh list.
REDO diskN: stalled … That device stopped accepting data and was given up on. Try it again alone; if it stalls again, throw it away.
REDO diskN: copy failed … The device may be faulty. Try a different hub port, or discard it.
A device is dark (no light) Its battery died or it switched off mid-copy, so it failed — but it can't blink. Charge it fully first (in the sun or with a proper charging cable — the data cable/hub may not charge it), then reconnect and redo it. (A steady/solid light means it finished; still verify its content.)
Stopped with Ctrl+C? The script lists which devices are INCOMPLETE. Do not hand those out — run again to redo them.
No disks listed Make sure the hub is connected and powered, and the devices are fully inserted.
A red-icon alert pops up The script raises an on-screen alert for the show-stoppers — no external devices found, or none could be prepared. It tells you what to check; fix it and run again.
Advanced (not generally recommended): loading from a device instead of a folder on your Mac. You can run the loader straight from a “master” device — a player, a thumb drive, etc. — by putting load_content.sh and your content folders on it and dragging that copy onto Terminal. If you do:
  • Plug the master into its own port on the Mac — a spare USB / USB-C port on the computer itself, not a spare socket on the same hub or adapter as the devices you're loading. That keeps it out of the shared-hub traffic (so the copy stays fast) and keeps its identity stable while devices are plugged and unplugged on the hub.
  • The loader protects the master automatically. It works out which physical disk the script is running from and removes that disk from everything it can erase — you'll see Skipping diskN — it holds the source content — and it only ever reads from the master, never formatting, writing to, or renaming it.
  • Still play it safe: the master is protected from erasure but is not hardware write-locked, so use a spare copy as the master rather than your only good device, and always check a freshly loaded device in the real player before doing a big batch.

Player-Specific Setup

This loader tool is player-agnostic: all it does is make a clean, full-size FAT32 volume and copy your numbered folders onto it in play order. It does not know or care which brand of player you own. This tab is about matching your source folder and your hardware to your specific player so that what the tool copies is what the player will actually read.

Will my player work?

A player is loadable with this tool only if all four of these are true:

  1. It shows up as a writable USB disk (or its card does). Either the player itself mounts on the Mac as an erasable removable drive, or it takes a removable microSD/SD card that you can pull and mount in a USB card reader.
  2. It reads a FAT32 volume. This tool formats FAT32; a player that demands exFAT or some other layout will not read what you produce.
  3. It plays plain, unencrypted audio files (normally MP3) that you organize into folders. Players that require a proprietary or encrypted container will ignore ordinary MP3s.
  4. It plays in folder/track order (either the copy order this tool writes, or a name-sort your numbered folders satisfy).
Sealed player, removable card? Many players have factory-sealed internal memory you cannot touch and a removable microSD card. In that case you load the card, not the player: pull the card, put it in a USB card reader, and run the tool against the card. A standalone reader is best — then only the one clean card volume mounts and you cannot erase the wrong thing.
Always test one device first. Load a single card or player, put it in the actual player, and confirm it plays in the right order before you run a batch. Several details below are marked unverified precisely because they can only be settled by testing one real unit.

MegaVoice

Solar/rechargeable audio Bible players. The critical fact: MegaVoice players' internal memory is not a plain USB drive — it is reachable only through MegaVoice's proprietary SLS cable plus their SaberCopy software. This tool cannot load internal memory. Where a MegaVoice player has a removable microSD card, that card is a plain FAT32 volume and the tool loads it fine.

General guidance for any player

Whatever the player, structure your source folder like this before running the tool:

  1. Number your top-level folders: 001-…, 002-…, 003-…, one per unit of content, in the order you want them played.
  2. Number your tracks with leading zeros inside each folder: 01, 02, … 10, 11 — so that 10 does not sort before 2.
  3. Use one audio format the device supports — usually MP3. Don't mix in formats a cheap player might not decode.
  4. Keep the nesting shallow. Match the depth your specific player expects (2 levels for the KULUMI Sheep, 3 for the KULUMI X/Mini, 4 for card-based MegaVoice); deeper or more elaborate trees confuse minimal firmware.
  5. Test one loaded device in the actual player before you run a batch. Confirm it powers on, finds the content, and plays in the order you intended. Only then load the rest.
Why the numbering matters: many of these players play in the order files were copied, not a firmware name-sort. This tool copies your folders in sorted name order onto a fresh volume, so numbered names with leading zeros are what put the tracks in the right sequence.
